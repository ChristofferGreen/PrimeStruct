Vector/map de-builtinization roadmap note: the target state is that, aside from minimal runtime/memory substrate, C++ compiler/backend code does not know collection semantics for `vector` or `map`; that behavior should live in stdlib `.prime` files.
SoA de-builtinization roadmap note: the target state for `soa_vector` is also a stdlib `.prime` implementation. Any temporary compiler/runtime support should be reduced to generic substrate only: field-layout/codegen/introspection, column storage, field-view borrowing/invalidation, and allocation primitives that do not mention `soa_vector` in C++ source. The public `/std/collections/soa_vector/*` behavior should eventually live entirely in `.prime`.

Execution order note: use this sequence and resist bridge churn outside it.
1. Finish only the runtime/ownership substrate that self-contained `.prime` containers genuinely need.
2. Keep landing real `.prime` container behavior under the experimental namespaces until VM/native/C++ parity is good enough to trust the implementations.
3. Replace wrapper-only canonical `/std/collections/vector/*` and `/std/collections/map/*` entry points with those real `.prime` implementations.
4. Delete compiler-owned compatibility handling and helper routing from semantics first, then IR lowering, then emitters/backends.
5. Leave only allocation/error/drop substrate in C++.
